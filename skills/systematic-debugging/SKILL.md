# 系统化调试

## 总览

随机修补会浪费时间并制造新 bug。快速补丁往往掩盖底层问题。

**核心原则：**永远先找根因，再尝试修复。修症状就是失败。

## 铁律

```text
没有根因调查，就不能修复
```

未完成阶段 1 前，不得提出修复方案。

## 何时使用

任何技术问题都使用：

- 测试失败。
- 生产 bug。
- 非预期行为。
- 性能问题。
- 构建失败。
- 集成问题。

尤其在以下情况必须使用：

- 时间压力大，容易猜。
- “就一个快速修复”看起来很明显。
- 已经尝试过多个修复。
- 上一个修复没生效。
- 你并未完全理解问题。

## 四个阶段

必须完成当前阶段，才能进入下一阶段。

### 阶段 1：根因调查

修复前必须：

1. **仔细读错误信息**
   - 不跳过错误或警告。
   - 完整读 stack trace。
   - 记录行号、文件路径、错误码。

2. **稳定复现**
   - 能否可靠触发？
   - 精确步骤是什么？
   - 是否每次发生？
   - 不能复现就继续收集数据，不猜。

3. **检查近期变化**
   - 哪些变化可能导致它？
   - 看 git diff、近期提交。
   - 新依赖、配置变化、环境差异。

4. **多组件系统收集证据**

   在 CI -> 构建 -> 签名、API -> 服务 -> 数据库这类链路中，先在每个边界加诊断：

   ```text
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置是否传递
     - 检查每层状态
   ```

   跑一次收集证据，确认是哪一层断，再调查那一层。

5. **追踪数据流**

   错误在深层调用栈时，阅读 `root-cause-tracing.md`。从坏值出现的位置一路向上追，找到来源，在来源修，不在症状处修。

### 阶段 2：模式分析

修复前先找模式：

1. 找同代码库中类似且工作的实现。
2. 若实现某模式，完整阅读参考实现，不要扫读。
3. 列出工作版本和坏版本的差异，别假设“小差异不重要”。
4. 理解依赖：配置、环境、组件、前提假设。

### 阶段 3：假设和验证

按科学方法：

1. 写出单一假设：“我认为根因是 X，因为 Y。”
2. 做最小改动验证假设，一次只改一个变量。
3. 验证结果。成功则进入阶段 4；失败则形成新假设。
4. 不懂就说“不理解 X”，继续研究或求助。

### 阶段 4：实现修复

修根因，不修症状：

1. **创建失败测试**：最小复现，优先自动化测试，没有框架就写一次性脚本。写测试时使用 `test-driven-development`。
2. **实现单一修复**：只处理已确认根因，不做“顺手改进”。
3. **验证修复**：测试通过，相关测试未坏，原问题确实消失。
4. **修复不生效时停止**：尝试少于 3 次则回到阶段 1；达到 3 次后停下来质疑架构。

### 阶段 4.5：三次修复失败后质疑架构

如果每次修复都暴露不同共享状态、耦合或新症状，或者修复需要大规模重构，可能是架构模式本身有问题。停止继续补丁，和用户讨论是否应重构架构。

## 红旗

出现这些想法就停下，回到阶段 1：

- “先快速修一下，之后再查。”
- “改 X 试试看。”
- “一次改几个地方跑测试。”
- “跳过测试，手工验证就行。”
- “大概是 X，我先修。”
- “我不完全懂，但也许能行。”
- “模式是 X，但我按自己的方式适配。”
- 已经尝试两次后还想“再试一次”。

## 用户信号

如果用户说：

- “这没发生吗？”说明你在假设。
- “这能显示什么？”说明你应该先加证据。
- “别猜了。”说明你在没理解前提方案。
- “仔细想。”说明要质疑基本假设，而不仅是症状。
