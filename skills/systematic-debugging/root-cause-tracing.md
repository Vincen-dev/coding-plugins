# 根因追踪

当错误出现在调用栈深处时，不要在报错点修症状。沿数据流向上追，直到找到坏值、错误状态或错误假设第一次出现的位置。

## 何时使用

- 错误发生在底层函数，但原因来自上层输入。
- 栈追踪很长，不清楚哪个调用者传错值。
- 修报错点只能掩盖问题。
- 同类错误在多个地方出现。

## 方法

1. **定位症状点**：哪里抛错、断言失败或输出错误？
2. **记录坏值**：坏值是什么？它和期望差在哪里？
3. **找直接来源**：谁把这个值传进来？
4. **继续向上追**：上一层的输入是否已经坏了？
5. **找到首次变坏点**：这里才是根因候选。
6. **解释机制**：为什么这里会产生坏值？
7. **在源头修复**：不要在下游兜底掩盖。

## 追踪模板

```text
Symptom:
- file:line
- observed:
- expected:

Trace:
1. Function A receives bad value X from Function B
2. Function B constructs X from Y
3. Y comes from config Z
4. Z is missing because ...

Root cause:
- ...

Fix location:
- ...
```

## 常见错误

- 在抛错点加默认值，但不查为什么缺值。
- 看到 null 就加 null check，但不查谁传 null。
- 修改测试预期而不查行为为什么变了。
- 只看最后一个 stack frame。

## 完成标准

你能回答：

- 坏值最早在哪里出现？
- 它为什么会出现？
- 修复为什么放在那里？
- 哪个测试能证明该路径已覆盖？
