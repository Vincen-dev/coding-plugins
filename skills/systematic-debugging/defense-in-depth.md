# 防御式修复

根因修复解决“为什么发生”。防御式修复解决“如果类似问题再出现，如何更早、更清楚、更安全地失败”。

## 原则

先修根因，再加防线。不能用防线代替根因修复。

## 常见防线

- 输入校验：在边界检查非法数据。
- 明确错误：报出可诊断信息，而不是静默失败。
- 类型收窄：让非法状态无法表示。
- 契约测试：覆盖模块边界。
- 断言：在不变量被破坏时尽早失败。
- 日志：记录跨组件数据流和配置传递。

## 例子

问题：下游函数收到空 ID 后查询错误记录。

错误修复：

```text
下游看到空 ID 就 return null
```

正确做法：

1. 查清空 ID 是谁生成的。
2. 在源头修复 ID 构造。
3. 在边界加校验：空 ID 直接抛出带上下文的错误。
4. 加测试覆盖源头和边界。

## 何时不加防线

- 只是为了隐藏错误。
- 让系统继续运行但数据已不可信。
- 防线引入复杂度大于风险。
- 没有先定位根因。

## 完成标准

- 根因已修。
- 防线有明确目的。
- 错误信息能帮助定位。
- 测试覆盖根因和防线。
