# 紧急 Bugfix 压力场景

紧急修复更需要 TDD，因为压力下最容易修错问题或造成回归。

## 操作顺序

1. 先复现 bug，记录输入、环境、当前错误。
2. 把复现路径写成失败测试。
3. 运行测试，确认 RED 失败 和真实 bug 一致。
4. 写最小修复。
5. 重跑 bug 复现测试和相关回归测试。
6. 如涉及根因不明，先使用 `systematic-debugging`，再进入 TDD 修复。
7. 回报 TDD 证据。

## RED 必须证明

- 测试失败信息能指向原 bug。
- 失败不是测试代码错误、导入错误或环境错误。
- 测试在修复前失败，在修复后通过。

## 不能自动化时

如果 bug 依赖外部系统、账号权限、真实设备或不可控时间窗口，先记录 TDD 例外记录，并提供替代验证：

- 复现步骤。
- 修复前后日志。
- 截图或录屏。
- 最小手工验收步骤。
- 后续自动化补测计划。
