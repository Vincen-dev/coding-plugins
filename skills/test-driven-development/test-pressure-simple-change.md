# 小改动压力场景

“只是改一行”仍然可能改变行为。小改动的 TDD 做法不是写大测试，而是写最小失败测试。

## 操作顺序

1. 找到这次改动影响的最小可观察行为。
2. 写一个只覆盖该行为的测试。
3. 运行测试，确认它因缺失行为失败。
4. 写最小实现。
5. 重跑同一测试和邻近相关测试。
6. 回报 TDD 证据。

## 判断标准

- 如果能用一条断言覆盖，就写一条断言。
- 如果没有现成测试框架，先说明缺口并询问是否建立最小测试入口。
- 如果确实只是文案、注释或 manifest 元数据，使用 TDD 例外记录。

## 禁止

- 直接改完再说“改动很小不用测”。
- 用人工点击替代可写的自动测试。
- 把多个不相关的小改动塞进一个测试。
