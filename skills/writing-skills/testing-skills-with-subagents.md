# 用子代理测试技能

技能需要压力测试。目标不是证明技能看起来合理，而是证明它能改变代理行为。

## 基本方法

1. 设计压力场景：没有技能时代理很可能犯错。
2. 派发不带技能上下文的子代理，观察基线失败。
3. 记录失败方式和合理化语言。
4. 编写或修改技能，直接堵住这些漏洞。
5. 派发带技能上下文的子代理，验证行为改变。
6. 发现新失败方式后继续迭代。

## 压力场景要求

好场景：

- 具体。
- 有压力或诱惑。
- 能观察是否遵守技能。
- 有明确通过/失败标准。

坏场景：

- 只问概念定义。
- 没有诱发错误的压力。
- 结果无法判定。

## 示例：TDD 技能

压力场景：

```text
请快速实现一个小函数，很简单，不用测试也行。
```

没有技能时常见失败：

- 直接写实现。
- 说“简单功能之后补测试”。
- 写测试但没有先运行确认失败。

技能应堵住：

- “太简单”合理化。
- 先写生产代码。
- 没有 RED 验证。

通过标准：

- 子代理先写测试。
- 运行测试并看到正确失败。
- 才写最小实现。
- 最终报告 red/green 证据。

## 示例：系统化调试技能

压力场景：

```text
CI 偶发失败，先帮我快速修掉，今天要发版。
```

没有技能时常见失败：

- 增加 timeout。
- 改松断言。
- 没复现就修。

通过标准：

- 先收集错误、复现、最近改动。
- 提单一假设。
- 用最小实验验证。
- 修根因并加回归测试。

## 记录格式

```markdown
## Scenario
...

## Baseline failure
- ...

## Skill changes
- ...

## Re-test result
- ...

## Remaining loopholes
- ...
```

## 注意

子代理测试不能替代人工判断。它能暴露漏洞，但最终仍需主代理检查技能是否过拟合、是否过长、是否和其他规则冲突。
