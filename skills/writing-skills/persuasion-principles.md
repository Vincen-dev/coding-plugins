# 技能中的说服原则

技能不是单纯命令列表。它要让未来代理在压力下仍然愿意遵守流程。

## 解释为什么

规则后面应说明原因。没有原因的规则容易被“这次例外”绕过。

坏：

```text
不要跳过测试。
```

好：

```text
不要跳过测试。没有先失败的测试，你不知道测试是否能抓住错误；之后补测试会被实现偏见污染。
```

## 点名合理化

代理常会合理化：

- “这个太简单。”
- “先快速看一下。”
- “之后补。”
- “用户很急。”
- “我已经知道怎么修。”

技能应明确这些想法是红旗，并给出替代行动。

## 使用硬门禁

对高风险流程使用门禁：

- 未完成根因调查，不得修复。
- 未看到失败测试，不得写生产代码。
- 未运行验证，不得声称完成。

门禁要少而硬。太多门禁会降低可执行性。

## 给替代行为

只说“不要做 X”不够，要说明“改做 Y”。

例如：

- 不要固定 sleep；等待明确条件。
- 不要泛泛评审；给 file:line 和严重级别。
- 不要表演式赞同；验证后修复或技术性反驳。

## 校准严重程度

技能应避免把所有问题都说成致命。真实严重级别能提高可信度：

- Critical：必须修，否则功能、安全或数据会坏。
- Important：应修，否则质量或维护性明显下降。
- Minor：可选改进。

## 保持人类控制权

技能约束“怎么做”，用户决定“做什么”。当用户明确指令、仓库规则或安全要求与技能冲突时，用户规则优先。
