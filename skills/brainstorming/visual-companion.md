# 视觉伴侣

视觉伴侣用于在脑暴阶段展示 mockup、线框图、图表、布局对比和视觉选项。它不是必选模式，而是一个在视觉内容更容易说明问题时使用的工具。

## 使用原则

- 只有用户同意后才能使用。
- 征求同意必须是单独消息，不和其他问题混合。
- 每个问题单独判断是否需要视觉化。
- 视觉内容应帮助决策，而不是装饰。

## 何时使用浏览器

适合：

- UI mockup。
- 布局或导航结构对比。
- 信息架构图。
- 流程图、状态图、组件关系图。
- 多个视觉方案并排比较。

不适合：

- 纯需求澄清。
- 文本选项 A/B/C。
- 技术取舍说明。
- 范围和优先级问题。

判断标准：用户看到它是否比读文字更容易理解。

## 征求同意话术

```text
我们接下来讨论的部分内容可能用浏览器展示更容易理解。我可以边讨论边做 mockup、图表、对比和其他视觉材料。这个功能还比较新，也会消耗较多 token。想试试吗？（需要打开一个本地 URL）
```

只发送这段，不要附加其他内容。用户拒绝时继续文本脑暴。

## 本地服务模式

原版提供了 `scripts/` 下的本地服务和 HTML frame 模板：

- `start-server.sh`
- `stop-server.sh`
- `server.cjs`
- `frame-template.html`
- `helper.js`

使用这些脚本时：

1. 启动本地服务。
2. 为当前问题生成一个清晰页面。
3. 给用户本地 URL。
4. 用户反馈后修改视觉材料。
5. 脑暴结束后停止服务。

## 视觉材料要求

- 一屏只服务一个决策。
- 标题和选项清楚。
- 不要做营销页或装饰性 hero。
- 不要用无意义渐变、光斑或占位图。
- UI mockup 要尽量接近真实尺寸和交互结构。
- 对比方案应统一尺度，避免视觉偏差。

## 输出方式

视觉问题可以这样推进：

```text
我会把这两个布局方案放到浏览器里对比：左侧是分步 wizard，右侧是单页表单。看完后我们只决定一个问题：哪个更适合你的主要用户？
```

不要把视觉伴侣变成无休止设计探索。每个视觉产物都应推进一个具体决策。

## 结束条件

- 用户已经做出视觉决策。
- 当前问题转为纯文本讨论。
- 脑暴阶段完成并进入规格文档。

结束时停止本地服务，避免留下后台进程。
