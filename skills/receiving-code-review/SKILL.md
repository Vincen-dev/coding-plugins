# 接收代码评审

## 总览

代码评审需要技术判断，不需要情绪表演。

**核心原则：**实现前先验证。不清楚先问。技术正确性高于社交舒适。

## 响应模式

收到反馈时：

1. **READ**：完整阅读反馈，不立即反应。
2. **UNDERSTAND**：用自己的话复述要求，或提问。
3. **VERIFY**：对照代码库现实验证。
4. **EVALUATE**：对这个代码库技术上是否成立。
5. **RESPOND**：技术性确认或有理由反驳。
6. **IMPLEMENT**：一次处理一项，每项都测试。

## 禁止回应

不要写：

- “你说得完全对！”
- “好建议！”
- “我现在就实现。”（还没验证前）

改为：

- 复述技术要求。
- 提澄清问题。
- 如果错误，用技术理由反驳。
- 或直接开始工作，用结果说话。

## 反馈不清楚

只要有一项不清楚，就先停下澄清。多项反馈可能相关，部分理解会导致错误实现。

如果理解 1、2、3、6，但 4、5 不清楚，不要先实现 1、2、3、6。先说明：“我理解 1、2、3、6；4 和 5 需要澄清后再实现。”

## 来源处理

### 来自用户

- 可信，但仍要理解清楚。
- 范围不清就问。
- 不做表演式赞同。
- 直接行动或给技术性确认。

### 来自外部评审者

实现前检查：

1. 对当前代码库技术上是否正确？
2. 是否破坏现有功能？
3. 当前实现是否有原因？
4. 是否适用于所有平台/版本？
5. 评审者是否理解完整上下文？

如果建议看起来错，用技术理由反驳。如果无法轻易验证，说明需要什么信息，并问要调查、询问还是继续。

## YAGNI 检查

评审者要求“专业地实现某功能”时，先搜索代码库实际使用：

- 未使用：询问是否删除它（YAGNI）。
- 已使用：再正确实现。

## 实现顺序

多项反馈：

1. 先澄清所有不清楚项。
2. 阻塞问题优先：崩溃、安全、数据丢失。
3. 简单修复：拼写、导入。
4. 复杂修复：重构、逻辑。
5. 每项单独测试。
6. 验证无回归。

## 何时反驳

当建议会破坏现有功能、评审者缺上下文、违反 YAGNI、对技术栈不正确、存在兼容性理由，或和用户先前架构决策冲突时，应该反驳。

反驳要基于技术理由、具体问题、工作测试或代码证据。架构冲突时让用户决策。

## 反馈正确时

可以说：

- `Fixed. [简述改动]`
- `已验证该问题存在，修复在 [location]。`
- 或直接修复并展示结果。

不要长篇道歉，不要表演感谢。行动和证据更重要。

## GitHub 线程回复

回复 GitHub inline review comment 时，在对应 comment thread 中回复，不要发成顶层 PR comment。

## 底线

外部反馈是待评估建议，不是必须执行的命令。验证、质疑，然后实现。
