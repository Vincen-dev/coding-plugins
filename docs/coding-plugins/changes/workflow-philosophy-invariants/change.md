# 工作流哲学不变量

## 意图

让测试先行、可验证契约、系统化执行、简单优先和证据先于声明成为所有活跃工作流必须遵守的不变量。

## 风险

本次变更影响所有 Agent-facing 工作流表面。主要风险是路由含糊、质量门槛意外丢失，以及退役术语清理不完整。

## 范围

- 范围内：活跃 Skills、prompts、Change Capsule 模板、维护中的用户文档、静态契约测试和退役工作流文档。
- 范围外：改写发布历史或保留可执行兼容层。
- 影响系统：`skills/`、`tests/ts/`、`docs/coding-plugins/`、README、安装与迁移说明。

## 可验证契约

- [x] VC-001：所有生产变更在实现前声明结果、边界和验证方式。
- [x] VC-002：所有行为变更在实现前观察失败测试或可复现失败检查；重构先建立通过的特征测试基线。
- [x] VC-003：调查和执行遵循明确顺序；缺少证据时停止，不进行猜测。
- [x] VC-004：活跃工作流只有一个状态源、按风险递进的产物，并且不依赖退役词汇或运行时。
- [x] VC-005：完成声明必须有最新验证输出和明确的剩余风险说明。

## 产物

- `change.md`：整个变更的唯一状态源。
- `plan.md`：可执行实施计划。
- `evidence.md`：测试先行和最终验证的实际证据。

## 批准记录

- 2026-07-11 范围/计划：用户要求清除所有旧契约，并按前述哲学分析进行优化。
- 2026-07-11 执行：同一指令明确要求实施并完成优化。

## 当前任务

已完成。不变量测试、活跃表面扫描和仓库全量验证均为绿色。

## 决策

- 活跃表面只使用 `Verifiable Contract`、`plan.md` 和 `evidence.md` 模型。
- Release Notes 和迁移指南只能把退役行为作为历史背景描述。
- 不重新引入 runtime 或隐藏项目状态来强制哲学规则。

## 完成情况

- 已实现：通用不变量、可验证契约模型、确定性阶段门禁、Quick evidence 契约、活跃旧语义清理和维护文档迁移。
- 已验证：22/22 测试、活跃表面零匹配扫描、YAML/JSON 解析、退役路径断言和 diff 完整性。
- 延后项：发布历史中的术语按历史记录保留。
- 剩余风险：纯工作流依赖 Agent 遵循 Skills；静态测试能够防止本仓库契约漂移，但不会自动审计外部业务仓库。
